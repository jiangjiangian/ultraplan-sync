## ⏱ 2:20–4:00　物件模型：GameObject + 角色介面(ISP) + CRTP（不用 dynamic_cast）

【操作】搜尋框打 `Roles` → 點 `Roles.h` 節點 → 細節面板按「📖 逐檔詳盡說明」。
深連結：<https://jiangjiangian.github.io/ultraplan-sync/#node=file:include/engine/core/Roles.h>

【說】
> 「地圖上每個會動或不會動的東西都是一個 `GameObject`。一開始我把它做成一個**肥大介面**——每個物件都被迫實作 `Update / Render / Interact`，連空殼都要寫，這很糟。
> 重構後我用**介面隔離原則（ISP）**把它拆成四個**單方法角色介面**：`IUpdatable`、`IDrawable`、`IInteractable`、`IMortal`。各類別只扮演自己需要的角色——消耗品只有 `IInteractable`、雨傘沒有 `IMortal`、玩家沒有 `IInteractable`。」

【操作】搜尋 `GameObject` → 看它的繼承樹（左側切「繼承樹」概念或看 `inherits` 邊）。
深連結：<https://jiangjiangian.github.io/ultraplan-sync/#node=file:include/engine/core/GameObject.h>

【說（重點，TA 會在意）】
> 「關鍵是中間這層 CRTP mixin `WithRoles<Derived, Base>`。我**明確要求不要用 `dynamic_cast`／RTTI**，要快。
> 它用編譯期的 `if constexpr + std::derived_from` 判斷某型別到底扮演哪些角色：扮演就回傳 `static_cast<Derived*>` 的 typed pointer，沒扮演就回 `nullptr`——**整個『查角色』是編譯期就決定的零成本操作**。
> 呼叫端只寫 `if (auto* u = o.AsUpdatable()) u->Update(dt);`——異質容器裡，誰能更新誰就更新，安全又零開銷。這是我這份作業最得意的一招『靜態多型取代動態轉型』。」

# 📚 設計模式與 SOLID 深入解說（Q&A／口試備答區）

> 這一區不在 10 分鐘內，是給口試追問用的「深水區」。每個模式都給：**是什麼 → 落點檔案 → 實作細節 → 為什麼選它（替代方案與取捨）→ 可能被問**。

## 一、四個核心 GoF 設計模式

### 1) State（狀態模式）
- **是什麼**：把「一個物件的行為隨內部狀態改變」封裝成獨立的狀態類別，狀態轉換 = 換一個狀態物件。
- **落點**：[`include/game/state/SemesterStateMachine.h`](https://github.com/jiangjiangian/ultraplan-sync/blob/main/graph/wiki/files/include__game__state__SemesterStateMachine.h.md)、`IChapterState` 介面、五個具體章節（`Chapter1AddDrop` … `Chapter4Finals`、`InterludeMarket`）。
- **實作細節**：機器持有 `std::unique_ptr<IChapterState>`；`Transition(next)` 析構舊狀態（呼叫 `Exit()`）、建構新狀態（呼叫 `Enter()`）。**幕間市集是共用轉運站**，被重複進出三次，下一站由 `InterludeReturnTo` 決定。結局**不是** `IChapterState` 子類別，而是用哨兵旗標記錄（見 `EndingGate`）。
- **為什麼選它（取捨）**：替代方案是「一個 `enum SemesterState` + 到處 switch」。我不選那個，因為每章的 `Enter/Update/Exit` 行為不同，散落的 switch 會在每加一章時被迫到處改、容易漏；用狀態物件把每章的行為**內聚**在自己的類別裡，新增章節＝加一個類別，符合 OCP。
- **可能被問**：「為什麼結局不做成狀態？」→ 結局是**終止態 + 判定結果**，沒有持續的每幀行為，用哨兵旗標 + `CheckEndingGates()` 集中判定比硬塞五個空狀態類別更誠實、更好讀。

### 2) Factory Method（工廠方法）
- **是什麼**：把「決定要 new 哪個具體類別」這件事，從呼叫端抽出來集中到一個生成函式。
- **落點**：[`include/game/controller/GameObjectFactory.h`](https://github.com/jiangjiangian/ultraplan-sync/blob/main/graph/wiki/files/include__game__controller__GameObjectFactory.h.md) + `src/game/controller/GameObjectFactory.cpp`。
- **實作細節**：`Create(ObjectType, Vec2)` 依 `ObjectType` 列舉回傳 `std::unique_ptr<GameObject>`；12 種具體型別對呼叫端（spawn 設定、`SpawnSystem`、章節 roster）**完全不可見**——它們只給一個 enum 值與座標。
- **為什麼選它（取捨）**：替代方案是呼叫端自己 `new CursedUmbrella(...)`。那會讓「知道所有具體傘類」這個知識**散落各處**；集中到工廠後，新增一種物件只動工廠一處（OCP），而且呼叫端不必 `#include` 任何具體類別（配合下面的 DIP）。
- **可能被問**：「這算 Factory Method 還是 Simple Factory？」→ 誠實說：它是「列舉驅動的集中生成」，精神上是 Factory Method 的**意圖**（封裝生成、對擴展開放），實作上用一個 switch 而非多個工廠子類別，因為產品種類固定且生成邏輯簡單，過度設計成一堆工廠子類別反而沒必要。

### 3) Template Method（範本方法）
- **是什麼**：基底定義演算法骨架，把其中**可變的步驟**宣告成純虛擬，留給子類別覆寫。
- **落點**：傘家族 `TransparentUmbrella::BeClaimed()`（純虛）、消耗品 `ConsumableItem::Consume()`（純虛）。
  [`TransparentUmbrella.h`](https://github.com/jiangjiangian/ultraplan-sync/blob/main/graph/wiki/files/include__game__entities__TransparentUmbrella.h.md)
- **實作細節**：基底的 `Interact()/OnPickup()` 是**固定流程**：偵測 → 過任務閘 → 入袋/撐傘 → 廣播事件；中間呼叫純虛 `BeClaimed(player)`。五把傘各自覆寫：`TrueUmbrella` 設真結局旗標、`CursedUmbrella` `IncCursedTaint()`、`FragileUmbrella` 最小副作用……
- **為什麼選它（取捨）**：替代方案是在基底用一個大 `switch(kind)`。Template Method 讓「共同流程只寫一次、差異各自封裝」，加一把新傘＝寫一個子類別覆寫一個方法，不必改基底——又是 OCP。
- **可能被問**：「和 Strategy 差在哪？」→ Template Method 用**繼承**在編譯期綁定可變步驟（傘的種類是 is-a）；Strategy 用**組合**在執行期換策略。傘的行為與身份綁定，用繼承（Template Method）較自然。

### 4) Observer（觀察者）
- **是什麼**：一對多的發布／訂閱，Subject 狀態改變時通知所有 Observer，雙方不直接相依。
- **落點**：[`include/engine/events/EventBus.h`](https://github.com/jiangjiangian/ultraplan-sync/blob/main/graph/wiki/files/include__engine__events__EventBus.h.md)（同時是 Singleton）。
- **實作細節**：`Publish(Event)` 前先用 `shared_lock` 取 handler 清單**快照**再廣播（避免廣播中有人退訂造成迭代失效）；`ScopedSubscribe` 回傳一個 **RAII `Subscription` 權杖**，出作用域自動退訂，杜絕懸空 handler。事件型別有 `UmbrellaClaimed / KarmaChanged / ShowMessage / EnteredBuilding / PickupAcquired`。
- **為什麼選它（取捨）**：替代是讓傘直接呼叫 `hud.showMessage(...)`。那會讓 Model 反向相依 UI、破壞分層。用 EventBus 後，傘只說「我被認領了」，誰要反應、怎麼反應，是訂閱者的事——Model 與 View 徹底解耦。
- **可能被問**：「Singleton 不是反模式嗎？怎麼測？」→ 我承認 Singleton 對測試不友善，所以留了 `EventSink` 接縫，測試可注入區域匯流排替換全域實例；`shared_mutex` 只保護 handler 清單，handler 本體仍須單執行緒呼叫（GL 限制），這點我在註解誠實標註。

## 二、另外三個結構性模式（加分題）
- **Strategy / Pipeline（策略管線）**：`ISystem::Run` 的五個 stage 由 `GameController` 依序執行——把 793 行 god method 拆成可組合、可重排、可單測的階段。[`SimSystem.h`](https://jiangjiangian.github.io/ultraplan-sync/#node=file:include/game/controller/SimSystem.h)
- **Singleton**：`EventBus::Instance()` 全域唯一事件匯流排（如上，已標註測試取捨）。
- **Command／Table（資料化命令表）**：`QuestHookTable` 把約 14 個內嵌 `TryXxx` 互動鉤子改成一張**有序、自我守門的資料表**，新增任務＝加一行 `RegisterHook`（OCP 的極致）。[`QuestHookTable.h`](https://jiangjiangian.github.io/ultraplan-sync/#node=file:include/game/quest/QuestHookTable.h)

## 三、兩個進階 OO 技法（不是 GoF 但很關鍵）
- **ISP 角色介面**：`Roles.h` 把胖介面拆成 `IUpdatable / IDrawable / IInteractable / IMortal`，物件只實作扮演的角色。
- **CRTP 靜態 mixin**：`WithRoles<Derived, Base>` 用 `std::derived_from + if constexpr` 在**編譯期**實作 `As*()` 能力查詢，**零成本取代 `dynamic_cast`**。這是「用模板做靜態多型」對「用虛擬函式做動態多型」的刻意取捨——能在編譯期決定的就不要拖到執行期。

## 四、SOLID 五原則 × 本專案實證

| 字母 | 原則 | 一句定義 | 本專案落點（可當場打開佐證） |
| :-- | :-- | :-- | :-- |
| **S** | 單一職責 | 一個類別只有一個被修改的理由 | 793 行 god method 拆成 **5 個 `ISystem`** + 4 個輸入處理器 + `QuestHookTable` + `SceneRouter`；`World`＝資料、`View`＝繪圖、`GameController`＝協調。 |
| **O** | 開放封閉 | 對擴展開放、對修改封閉 | 新增物件＝改 `GameObjectFactory` 一處；新增任務＝`QuestHookTable` 加一行；新增模擬階段＝push 一個 `ISystem`——皆不動既有控制流。 |
| **L** | 里氏替換 | 子類別能完全替換父類別 | 五個 `IChapterState`、`Vendor : NPC` 皆可代換；異質容器只持 `GameObject*`，靠 `As*()` 角色存取安全分派。 |
| **I** | 介面隔離 | 不強迫依賴用不到的介面 | `Roles.h` 四個**單方法**角色介面；消耗品不必實作 `Update`、看板不必 `IMortal`。 |
| **D** | 依賴反轉 | 高低階都依賴抽象 | `Render` 只認 `IRenderer&`（具體只有 `RaylibRenderer`）；輸入經 `InputSource` 抽象，可在真鍵盤與腳本輸入間切換（這正是確定性重播的基礎）。 |

> **一句總結（口試結尾可用）**：這五個原則不是事後貼標籤——它們是我重構的**動機**：因為要 SRP 才拆管線、因為要 OCP 才有工廠與鉤子表、因為要 DIP 才有 `IRenderer` 與 `InputSource`、因為要 ISP 才把胖介面拆成角色、因為要 LSP 才有可代換的狀態與 NPC 子類。

## 🎯 口試常見問題・速答小抄

- **「哪一行證明 engine 不依賴 game？」**→ `grep -r '#include "game' include/engine/` 沒有結果；`World`/entity 不含 `raylib.h`。
- **「為什麼不用 `dynamic_cast`？」**→ 要零成本、不要 RTTI。`WithRoles` 用 `if constexpr + std::derived_from` 編譯期決定，退化成 `static_cast`。
- **「793 行怎麼拆的？」**→ 依「一件事一個 stage」拆成 Survival/Movement/Collision/Spawn/Sweep 五個 `ISystem`，用 `SimContext` 串接，Controller 只負責依序跑。
- **「四結局怎麼保證不 soft-lock？」**→ 旗標 `Flag_TaFinaleChoiceMade` 一旦設立即為 TOTAL，`CheckEndingGates` 每非對話幀以 A→B→D→C 優先序輪詢，四選一必中。
- **「測試怎麼測即時遊戲？」**→ `InputSource` 抽象 + 固定 1/60 timestep ⇒ 腳本重播逐位元一致，`CHECK(identical)`。
- **「smart pointer 用在哪？」**→ `World` 以 `std::unique_ptr<GameObject>` 持有所有物件（RAII、不漏記憶體）；移除採 `isActive_` 旗標 + 幀末 `Sweep()`，不在迭代中 `delete`。
